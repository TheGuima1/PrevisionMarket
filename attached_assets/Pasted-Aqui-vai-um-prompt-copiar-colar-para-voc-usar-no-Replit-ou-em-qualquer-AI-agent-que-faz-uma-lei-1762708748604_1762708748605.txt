# PROMPT (cole exatamente isto)

**Contexto**
Você é um analista técnico-didático. Seu trabalho agora é **explicar meu sistema de forma leiga**, como se estivesse ensinando para alguém que nunca programou, ligando cada parte do **código** ao que o **usuário vê e faz**.

**Escopo do que analisar**

* Ler todo o repositório (arquivos front-end, back-end, scripts e configs).
* Identificar tecnologias (frameworks, bibliotecas, serviços externos).
* Mapear telas/fluxos do usuário e como o código implementa cada passo.
* Explicar onde os **dados** são guardados e como viajam entre front-end ↔ back-end ↔ banco/serviços.

**Estilo e linguagem**

* Português claro, **sem jargão**. Quando usar termos técnicos, explique entre parênteses.
* Frases curtas, exemplos práticos, analogias simples.
* Nada de “devêssemos”, “pipeline CI/CD”, etc. Diga: “o código que testa antes de publicar”, e assim por diante.

**ENTREGÁVEIS (nesta ordem):**

1. **Resumo em 1 minuto (não técnico)**

   * O que o sistema faz.
   * Para quem é.
   * Problema que resolve e resultado para o usuário.

2. **Mapa das Funções Principais (lista simples)**
   Para cada função/feature visível ao usuário (ex.: “Criar conta”, “Login com 2 cliques”, “Fazer depósito PIX”, “Ver odds”, “Confirmar aposta”, “Saque”):

   * O que o usuário faz na tela.
   * O que o sistema faz por trás (em 2–3 frases).
   * Arquivos e trechos de código envolvidos (passe caminhos como `frontend/pages/Deposit.tsx`, `api/routes/bets.post.ts`).

3. **Como o Código Vira Experiência (UX)**
   Explique, para as **3 jornadas mais importantes**, o passo a passo “usuário clica → código roda”:

   * **Jornada A**: [ex.: Novo usuário cria conta e faz primeiro depósito]

   * **Jornada B**: [ex.: Usuário faz sua primeira aposta]

   * **Jornada C**: [ex.: Usuário saca dinheiro]
     Para cada passo:

   * Ação do usuário (ex.: “clicar em Depositar”).

   * Reação visível (ex.: “aparece QR Code PIX”).

   * O que o código executa (ex.: “chama `/api/deposits` que cria cobrança e grava status ‘pendente’”).

   * Onde isso está no código (arquivo + função).

   > **Mostre também um mini diagrama em texto** (ASCII) de cada jornada, assim:

   ```
   Tela do Usuário → Front-end (componente X) → API (/rota) → Banco (tabela Y) → Resposta → Atualiza Tela
   ```

4. **Dados: Onde Guardamos e Como Circulam**

   * **Armazenamento**: banco(s) usados, tabelas/coleções principais e o que vai em cada uma.

   * **Fluxo de dados**: de onde vêm, para onde vão, quando são atualizados.

   * **Estado no navegador**: o que fica salvo no cliente (cookies, localStorage, state manager).

   * **Segurança básica**: logins, tokens/sessões, criptografia (explique em linguagem simples).

   > Incluir uma **tabela simples**:
   > | Dado | Onde nasce | Onde fica salvo | Quem pode ver | Por quanto tempo |
   > |------|------------|-----------------|---------------|------------------|

5. **Arquitetura em 1 página (sem jargão)**
   Uma explicação visual em texto (ASCII) mostrando camadas:

   ```
   [Usuário]
      ↓
   [Front-end: páginas e componentes]
      ↓ chama
   [Back-end: rotas/controllers/serviços]
      ↓ acessa
   [Banco de Dados / Armazenamento]
      ↔ integra
   [Serviços Externos: pagamentos/SDK/oráculos]
   ```

6. **Arquivo-por-Arquivo (tour guiado)**
   Liste as pastas/arquivos importantes e diga **em 1 linha** para que servem.
   Ex.:

   * `src/pages/Login.tsx`: tela de login; valida e envia credenciais.
   * `src/api/deposits.ts`: cria cobranças PIX e atualiza status no banco.
   * `prisma/schema.prisma`: define tabelas de usuários, transações e apostas.

7. **Pontos de UX que Afetam o Código (e vice-versa)**

   * Onde o design da tela exige regra de negócio específica (ex.: “mostrar odds atualizando a cada X segundos”).
   * Onde a arquitetura atual atrapalha a UX (ex.: “tela pisca porque a rota responde devagar”).
   * Sugestões simples para melhorar fluidez (ex.: “pré-carregar dados”, “loading claro”, “mensagens de erro humanas”).

8. **Erros Comuns e Mensagens Humanas**
   Liste os 8–10 erros mais prováveis (ex.: “PIX expirou”, “saldo insuficiente”) e **o texto amigável** que o usuário deveria ver, + qual arquivo trata isso hoje.

9. **Desempenho e Estabilidade (versão leiga)**

   * Onde pode ficar lento e como melhorar em linguagem simples (cache = “guardar um atalho”, fila = “linha de espera”).
   * Monitoramento que existe/não existe (logs básicos, alertas).
   * O que testar primeiro (testes de caminho feliz e de erro nas 3 jornadas).

10. **Integrações Externas (se houver)**

    * Quem são (pagamentos, carteiras, oráculos, e-mail, analytics).
    * O que trocam de informação com o sistema.
    * Onde isso aparece no código (arquivo + função).

11. **Guia de Execução Local e Deploy (passo a passo)**

    * Como rodar localmente do zero (instalar deps, variáveis de ambiente, comando para iniciar).
    * Como publicar (em termos simples: “enviar para servidor X”, “configurar chave Y”).
    * Checklist rápido de “pronto para produção”.

12. **Glossário Leigo (A→Z)**
    Explique termos que você usar (ex.: “endpoint”, “token”, “webhook”, “state”) em 1 frase simples.

13. **Prioridades (próximos passos em 7 dias)**
    Faça uma **lista de 5–8 ações objetivas**, ordenadas por impacto na experiência do usuário (ex.: “1. Melhorar feedback de carregamento na tela de Depósito”, “2. Salvar rascunho de aposta para não perder ao recarregar”).

**Regras finais**

* **Não** altere código; apenas explique.
* **Inclua caminhos de arquivos e nomes de funções** sempre que citar “o código que faz X”.
* Se algo estiver faltando no repo (ex.: variável `.env`), diga claramente o que falta e por quê.
* Se houver ambiguidade, faça uma suposição **explícita** e siga.

**Saída**
Entregue em **markdown**, com títulos, listas, tabelas e diagramas em texto.
